# Sprint Planning

## Inputs
- Product brief with prioritized backlog
- Previous sprint retro action items
- Design system health report (from system-health chain)
- Outstanding drift or deprecation items

## Planning process
1. Review sprint goal (PM owns)
2. Review DS health items that are overdue
3. Size stories (team)
4. Commit to sprint scope
5. Identify which chains each story triggers:
   - New component work → new-component chain
   - Updates → component-update chain
   - Accessibility fixes → a11y-remediation chain
   - Token changes → token-migration chain

## Sprint ceremonies
- **Planning**: Define scope, identify chains
- **Daily sync**: Progress against stories
- **Design review**: Mid-sprint, using critique framework
- **Demo**: Show working output against acceptance criteria
- **Retro**: What worked, what didn't, what to change
